# MIDI 設計と Windows WinMM の既知問題

## 設計方針

xdx-rs の MIDI 層は以下の3つの問題を解決することを目的として設計されている。

1. **GUI スレッドのブロック防止** — 大きな SysEx 送信（~1秒）で UI が固まる
2. **SysEx チャンク分割への対応** — WinMM が大きな SysEx を分割して届ける
3. **ポートスキャンのブロック防止** — WinMM 不正状態でスキャンが永久ブロックする

---

## MIDI OUT: ワーカースレッド設計

### 問題

WinMM の MIDI 送信 API は、バッファの送信完了を待機するため、
大きなデータ（32-voice SysEx = 4104 バイト ≒ 1秒）の送信中に
呼び出し元スレッドをブロックする。
GUI スレッドから直接呼ぶと UI が固まる。

### 設計

MIDI OUT の接続（`MidiOutputConnection`）はワーカースレッドが所有する。
GUI スレッドはチャンネル経由でバイト列を渡すのみで、即座に返却を受ける。
実際の MIDI 送信はワーカースレッド内で行われ、GUI とは無関係に完了する。

```
GUI スレッド                         ワーカースレッド
  [Send ボタン押下]
  バイト列を生成
  チャンネルにキュー ──────────────→  キューからデータ取得
  即返却（ノンブロッキング）            MIDI OUT へ送信（ブロック可）
                                      送信完了後、次を待機
  [接続を閉じる]
  チャンネルの送信端を drop ──────→  受信端が Err を返す
                                      自然終了、接続を閉じる
```

### 注意点: Send 後に接続を閉じてはいけない

`send()` はノンブロッキングでチャンネルにキューするだけである。
直後に接続を閉じると、ワーカーがまだ送信中にチャンネルが切断される。
その後の別操作が新たな接続を開こうとすると、旧ワーカーと競合して
同じ MIDI ポートを2スレッドが同時使用する状態になり WinMM が破壊される。

→ **Send 操作後は接続を閉じない。接続はアプリ終了時（`on_exit`）に閉じる。**

Fetch 操作は受信完了を確認してから接続を閉じるため、このリスクはない。

## MIDI IN: SysEx 累積バッファ

### 問題

WinMM は内部バッファサイズ単位（~1024 バイト）で MIDI コールバックを呼ぶ。
大きな SysEx（4104 バイト）は複数回のコールバックに分割して届くことがある。
最初のチャンクだけでデコードしようとすると `TooShort` エラーになる。

### 設計

MIDI IN コールバック内に累積バッファを持ち、受信したバイト列を蓄積する。
`F0`（SysEx 開始）が来たらバッファをリセットして蓄積開始。
継続チャンクはバッファに追加。
`F7`（SysEx 終端）がバッファの末尾に来たとき初めて完全な SysEx として上位に渡す。

```
コールバック呼び出し1: [F0 43 04 20 ... ] → バッファに蓄積（F7 なし）
コールバック呼び出し2: [ ... 続き ... ]   → バッファに追加（F7 なし）
  :
コールバック呼び出しN: [ ... F7 ]         → バッファに追加 → SysEx として発行
```

これにより、midir / WinMM のチャンク分割が上位レイヤーに透過的になる。

## ポートスキャン: バックグラウンドスレッド

### 問題

MIDI デバイスの列挙（`list_in_ports` / `list_out_ports`）は、
WinMM が不正状態（デッドロック）にあると永久にブロックする。
GUI スレッドで直接呼ぶと Settings メニューを開いただけで UI が固まる。

### 設計

ポートスキャンは常にバックグラウンドスレッドで実行し、結果をキャッシュする。
GUI スレッドはキャッシュされたリストを参照するだけで、直接スキャンしない。

```
[アプリ起動 / Scan Ports 押下]
  バックグラウンドスレッド起動
  GUI はキャッシュ済みリストを表示し続ける（ブロックしない）

バックグラウンドスレッド:
  list_in_ports(), list_out_ports() を実行（ブロックする可能性あり）
  完了したらチャンネルで結果を送信

GUI update() ループ:
  チャンネルをポーリング（ノンブロッキング）
  結果が届いたらキャッシュを更新
```

### タイムアウトと再試行

WinMM が完全にデッドロックしている場合、スキャンスレッドが永久ブロックする。
スキャン開始から 5 秒後にタイムアウトとみなし、スキャン中フラグを解除する。
放棄されたスレッドはプロセス終了時に回収される。

タイムアウト後は Scan Ports ボタンを再度押せるようにする。
ステータスバーには Windows Audio サービスの再起動コマンドを案内する。

## アプリ終了時のクリーンアップ

egui の `on_exit` フックで MIDI IN / OUT の接続を明示的に閉じる。
正常終了（✕ボタン）の場合はこれにより WinMM リソースが解放される。

`cargo run` 中の Ctrl+C はプロセスが強制終了されるため `on_exit` が呼ばれない。
この場合は WinMM の内部状態が残り、デバイスロックが発生する可能性がある。

## Windows WinMM の既知問題

### 問題 1: 大きな SysEx 送信中の GUI フリーズ

- **症状**: Send 32 ボタン押下後、約 1 秒間 UI が固まる
- **原因**: WinMM が送信バッファの完了を待機する間、呼び出しスレッドをブロックする
- **対策**: MIDI OUT ワーカースレッド設計（前述）

### 問題 2: Send 後の即時切断による競合

- **症状**: Send 32 後にアプリが使えなくなる、MIDI デバイスが破壊される
- **原因**: ノンブロッキング `send()` の直後に接続を閉じると、送信中のワーカーと
  新たに開いた接続が同じポートを同時使用する
- **対策**: Send 後は接続を閉じない（前述）

### 問題 3: ポートスキャンの永久ブロック

- **症状**: Settings メニューを開くと UI が固まる（MIDI 不正状態後）
- **原因**: ポート列挙 API が WinMM の内部ミューテックス取得で永久ブロックする
- **対策**: バックグラウンドスキャン + タイムアウト（前述）

### 問題 4: デバイスロック（アプリ強制終了後）

- **症状**: アプリ強制終了後、MIDI デバイスが使えなくなる
- **原因**: WinMM の内部ミューテックスが解放されずデッドロック状態になる。
  プロセスが死んでもカーネルレベルの状態が残る場合がある
- **対策（アプリ側）**: 正常終了時の `on_exit` クリーンアップ
- **対策（OS 側）**: 下記「復帰手順」参照

## MIDI 不具合発生時の復帰手順

### ステップ 1: アプリで再スキャン

Settings → **Scan Ports** を押す。5 秒以内にポートが表示されれば復帰。

### ステップ 2: デバイスマネージャーで再初期化

デバイスマネージャーで USB-MIDI デバイスを「無効」→「有効」に切り替える。
USB の抜き差しよりもドライバレベルで再初期化されるため有効なことがある。

### ステップ 3: Windows Audio サービスの再起動（管理者 PowerShell）

```
Stop-Service audiosrv -Force
Stop-Service AudioEndpointBuilder -Force
Start-Service AudioEndpointBuilder
Start-Service audiosrv
```

停止できない場合は AudioEndpointBuilder の svchost プロセスを強制終了する:

```
# PID 確認
(Get-WmiObject Win32_Service -Filter "Name='AudioEndpointBuilder'").ProcessId
# 強制終了（PID を置き換え）
Stop-Process -Id <PID> -Force
Start-Service AudioEndpointBuilder
Start-Service audiosrv
```

### ステップ 4: PC 再起動

上記で解決しない場合は再起動が唯一の確実な手段。

---

## 再発防止のためのガイドライン

- アプリは必ず **✕ボタン** で正常終了する
- `cargo run` 中の **Ctrl+C は使わない**（`on_exit` が呼ばれない）
- 開発中は `cargo build` でビルドし、生成された `.exe` を直接起動することを推奨
